# Query Keys

A **query key** uniquely identifies each query’s data in TanStack Query’s cache. It must be an array at the top level and can be as simple as a single string or as complex as an array containing multiple strings and objects.
As long as it’s serializable with `JSON.stringify` and **unique to the data**, you can use it.

## Simple Query Keys

Use simple query keys for general or list-type resources.

```ts
// A list of todos
useQuery({ queryKey: ["todos"], queryFn: fetchTodoList });

// Any other static resource
useQuery({ queryKey: ["something", "special"], queryFn: fetchSomething });
```

## Query Keys with Variables

When your query depends on parameters (like an ID, filter, or options),
you can include them in the query key to keep caching accurate and isolated.

```ts
// A single todo item
useQuery({ queryKey: ["todo", 5], queryFn: () => fetchTodoById(5) });

// A preview version of a todo
useQuery({
  queryKey: ["todo", 5, { preview: true }],
  queryFn: () => fetchTodoPreview(5),
});

// A list of todos that are marked as done
useQuery({ queryKey: ["todos", { type: "done" }], queryFn: fetchDoneTodos });
```

## Deterministic Hashing

TanStack Query **hashes keys deterministically**,
meaning `object property order doesn’t matter` - `but array order does`.

These are considered **equal**:

```ts
["todos", { status, page }],
["todos", { page, status }],
["todos", { page, status, other: undefined }],
```

These are **not equal**:

```ts
["todos", status, page],
["todos", page, status],
["todos", undefined, page, status],
```

## Including Variables in Query Keys

If your `queryFn` uses variables (like `id` or `filter`),
always include them in your `queryKey` so TanStack Query can manage caching correctly.

```ts
function TodoItem({ todoId }) {
  const { data } = useQuery({
    queryKey: ["todo", todoId],
    queryFn: () => fetchTodoById(todoId),
  });

  return <div>{data?.title}</div>;
}
```

This ensures that:

- Each `todoId` gets its own cache entry.
- Data automatically refetches when `todoId` changes.

## Rule of thumb

- Query keys describe **what data** is being fetched.
- If your data changes, your query key should too.
- Always prefer **arrays** over strings for consistency.

## Example Summary

| Type          | Example                              | Description             |
| ------------- | ------------------------------------ | ----------------------- |
| Simple key    | `["todos"]`                          | General data list       |
| With variable | `["todo", 5]`                        | Specific resource       |
| With params   | `["todos", { status: "done" }]`      | Filtered list           |
| Nested object | `["user", { id: 1, profile: true }]` | Complex data dependency |
